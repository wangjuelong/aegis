# 完整技术解决方案

> 对标 CrowdStrike Falcon / Microsoft Defender for Endpoint / SentinelOne Singularity，设计一套功能不妥协、架构可落地的完整 EDR 平台。
> 

## 一、总体定位与设计目标

| 维度 | 目标 |
| --- | --- |
| 覆盖范围 | Windows / Linux / macOS 全平台 Agent；支持物理机、虚拟机、容器（K8s Pod） |
| 检测能力 | 实时行为检测 + IOC 匹配 + ML/AI 异常检测 + 威胁狩猎（Threat Hunting） |
| 响应能力 | 远程隔离、进程终止、文件隔离、注册表回滚、实时取证、自动化剧本（Playbook） |
| 性能约束 | Agent CPU ≤ 2%（稳态）、内存 ≤ 150MB、网络带宽 ≤ 50KB/s（均值） |
| 延迟指标 | 端到端检测延迟 ≤ 1s（关键规则）、云端关联分析 ≤ 5s |
| 规模目标 | 单集群支持 100 万端点，日处理事件量 ≥ 500 亿条 |

## 三、Endpoint Agent 详细设计

### 3.1 Agent 分层架构

```
用户态 (Ring 3)
┌─────────────────────────────────────────────────────────────┐
│  Agent Orchestrator (主进程)                                 │
│  ├── Config Manager         — 策略热更新、灰度下发           │
│  ├── Telemetry Collector    — 归一化所有 Sensor 事件         │
│  ├── Local Detection Engine — 本地规则 + 轻量 ML 推理        │
│  ├── Response Executor      — 执行云端/本地下发的响应动作     │
│  ├── Comms Module           — gRPC 长连接 + 本地 WAL 缓存   │
│  └── Self-Protection        — Agent 反篡改、完整性校验       │
├─────────────────────────────────────────────────────────────┤
│  Sensor Plugins (可插拔)                                     │
│  ├── Process Sensor         — 进程创建/退出/注入/Hollowing   │
│  ├── File Sensor            — 文件 CRUD / 重命名 / 权限变更  │
│  ├── Network Sensor         — Socket/DNS/HTTP(S) 连接追踪    │
│  ├── Registry Sensor (Win)  — 注册表读写/自启动项监控        │
│  ├── Auth Sensor            — 登录/提权/Kerberos 票据        │
│  ├── Script Sensor          — PowerShell/WMI/Bash 命令捕获   │
│  ├── Memory Sensor          — 内存扫描 YARA / 反射注入检测   │
│  └── Container Sensor       — cgroup/namespace 感知          │
└─────────────────────────────────────────────────────────────┘

内核态 (Ring 0)
┌─────────────────────────────────────────────────────────────┐
│  Kernel Driver / eBPF Programs                               │
│  ├── ETW Provider (Win)      / kprobe+tracepoint (Linux)     │
│  ├── Minifilter (Win)        / fanotify (Linux)              │
│  ├── WFP Callout (Win)       / nftables hook (Linux)         │
│  ├── AMSI Hook (Win)         / LSM Hook (Linux)              │
│  └── Shared Ring Buffer      — 零拷贝事件传递到用户态        │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 Sensor 采集能力矩阵

| Sensor | Windows 实现 | Linux 实现 | macOS 实现 | 采集粒度 |
| --- | --- | --- | --- | --- |
| Process | ETW (Microsoft-Windows-Kernel-Process) + Minifilter | kprobe on `do_execve` / eBPF `sched_process_exec` | Endpoint Security Framework (ESF) | 进程树、命令行、环境变量、签名信息、PE/ELF 元数据 |
| File | Minifilter (IRP_MJ_CREATE/WRITE/SET_INFO) | fanotify / eBPF `vfs_write` | ESF `ES_EVENT_TYPE_NOTIFY_WRITE` | 路径、哈希(SHA256)、entropy、Magic bytes、所属进程 |
| Network | WFP Callout Driver + ETW | eBPF `tcp_connect` / `sock_sendmsg` | Network Extension | 五元组、DNS query/response、SNI、JA3/JA3S 指纹 |
| Registry | ETW + CmRegisterCallback | N/A | N/A | Key/Value 路径、操作类型、旧值/新值、操作进程 |
| Auth | Security Event Log (4624/4625/4672/4768) | PAM + audit | OpenDirectory + ESF | 登录类型、来源IP、特权提升、Kerberos TGT/TGS |
| Script | AMSI (AntiMalware Scan Interface) | bash PROMPT_COMMAND / eBPF `uretprobe` | ESF | 完整脚本内容（解混淆后）、解释器 PID |
| Memory | VirtualAlloc/NtMapViewOfSection Hook | `process_vm_readv` + YARA | mach_vm_read | 可疑内存区域 dump、注入检测、无文件攻击痕迹 |
| Container | — | eBPF + cgroup ID + namespace ID | — | Pod/Container 元数据、逃逸检测 |

### 3.3 本地检测引擎

Agent 内嵌一个轻量检测引擎，负责**毫秒级**的第一道防线，不依赖网络：

```
事件流 ──→ ┌─────────────────────────┐
           │ Stage 1: Bloom Filter   │ ← IOC 哈希快速过滤（SHA256 of files/IPs/domains）
           │ (O(1), 误报率 ≤ 0.01%)  │
           └──────────┬──────────────┘
                      ↓ 可疑
           ┌─────────────────────────┐
           │ Stage 2: Sigma/YARA     │ ← Sigma 规则编译为字节码 + YARA 内存扫描
           │ Rules (本地编译执行)     │
           └──────────┬──────────────┘
                      ↓ 可疑
           ┌─────────────────────────┐
           │ Stage 3: Lightweight ML │ ← 梯度提升树 (XGBoost) / 小型 CNN
           │ (ONNX Runtime, <20MB)   │   静态 PE/ELF 特征分析
           └──────────┬──────────────┘
                      ↓ 高置信
           ┌─────────────────────────┐
           │ Local Response Action   │ ← 根据策略自动 Kill/Quarantine/Alert
           └─────────────────────────┘
```

**规则更新机制：**

- 云端编译 Sigma → 内部字节码 IR → 签名后推送
- Agent 热加载，无需重启；规则版本化 + 回滚

### 3.4 通信与容错

| 特性 | 设计 |
| --- | --- |
| 协议 | gRPC over TLS 1.3 + mTLS（Agent 证书轮换） |
| 压缩 | LZ4 流式压缩，典型压缩比 5:1 |
| 离线缓冲 | 本地 WAL（Write-Ahead Log），AES-256-GCM 加密，容量上限 500MB |
| 重连策略 | 指数退避 + 抖动，最大间隔 5min |
| 心跳 | 30s 间隔，含 Agent 健康摘要（CPU/Mem/版本/策略版本） |
| 命令通道 | 双向 gRPC Stream，云端可即时下发响应指令 |

### 3.5 Agent 自保护

- **内核级保护**：Minifilter/LSM 阻止对 Agent 进程/文件/注册表键的篡改
- **代码签名**：Agent 二进制全链签名验证，驱动 WHQL/DKMS 签名
- **Secure Boot 集成**：UEFI Secure Boot 链验证内核驱动完整性
- **Watchdog**：独立看门狗进程 + 系统服务互相监控，被杀自动重启
- **Anti-Debug**：检测调试器附加、内存断点、动态分析环境

### 4.2 Kafka Topic 设计

> **规范性文档**：详见 `docs/architecture/aegis-transport-architecture.md` 第 4.4 节。本节仅保留概要，以 Transport 架构文档为唯一事实来源（Single Source of Truth）。

| Topic | 分区策略 | 保留期 | 用途 |
| --- | --- | --- | --- |
| `raw-events.{tenant}` | Agent ID hash | 72h | 原始遥测数据入口 |
| `enriched-events` | Event Type | 72h | 富化后事件，供流处理消费 |
| `detections` | Severity | 30d | 确认的检测告警 |
| `commands.unicast` | Agent ID | 24h | agent-scoped 下行命令（RESPONSE_ACTION、REMOTE_SHELL 等）；Gateway 依 Connection Registry 投递到 owner pod |
| `commands.broadcast` | Round-robin | 24h | tenant/global-scoped 下行命令（POLICY_UPDATE、RULE_UPDATE、IOC_UPDATE、CONFIG_CHANGE）；每个 Gateway pod 独立 consumer group 全量消费 |
| `commands.pending` | Agent ID | 7d（compact+delete）| 目标 Agent 离线或瞬时失败的命令补投队列，与 Consumer 位点提交在同一 Kafka 事务 |
| `commands.dead-letter` | Agent ID | 30d | TTL 过期或多次失败的命令归档，审计 + 告警 |
| `audit-log` | Tenant ID | 365d | 操作审计 |

> **历史说明**：旧草案中的 `responses` 与 `threat-intel-updates` topic 已分别被 `commands.unicast` 和 `commands.broadcast` 取代，不再单独存在；威胁情报纳入广播通道后通过 `command_type=IOC_UPDATE` 区分。

### 5.2 检测引擎分层模型

```
Layer 5: Campaign Intelligence
  │  跨组织的 APT 活动关联，MITRE ATT&CK Group 映射
  ▼
Layer 4: Kill Chain Correlation
  │  跨端点、跨时间窗口的攻击链条重建
  │  Reconnaissance → Initial Access → Execution → Persistence → ...
  ▼
Layer 3: Behavioral Analytics (ML/AI)
  │  行为序列建模、异常评分、对抗样本鲁棒性

  │  Sigma 规则、YARA 规则、自定义 DSL 规则

  ▼
Layer 1: Telemetry Collection & Enrichment
     原始事件采集 → 归一化 → GeoIP/Asset/User 富化
```

### 6.2 数据模型（核心 Schema）

采用统一事件模型，兼容 ECS (Elastic Common Schema) / OCSF (Open Cybersecurity Schema Framework)：

```
TelemetryEvent {
  event_id:        UUID          // 全局唯一
  timestamp:       int64         // 纳秒精度 Unix 时间戳
  tenant_id:       string        // 租户标识
  agent_id:        string        // Agent 唯一标识
  host: {
    hostname:      string
    os:            string        // "windows-11-22H2", "ubuntu-22.04"
    ip:            string[]
    mac:           string[]
    asset_tags:    string[]      // CMDB 资产标签
  }
  event_type:      enum          // PROCESS_CREATE | FILE_WRITE | NET_CONNECT | ...
  severity:        enum          // INFO | LOW | MEDIUM | HIGH | CRITICAL

  // 多态 payload，按 event_type 确定具体结构
  process?: {
    pid:           int32
    ppid:          int32
    name:          string
    cmdline:       string
    exe_path:      string
    exe_hash:      string        // SHA256
    user:          string
    integrity:     string        // System/High/Medium/Low
    signature:     SignatureInfo
    tree:          ProcessNode[] // 完整进程树上下文 (parent → grandparent)
  }
  file?: {
    path:          string
    hash:          string
    size:          int64
    entropy:       float
    magic:         string
    action:        enum          // CREATE | MODIFY | DELETE | RENAME
  }
  network?: {
    src_ip:        string
    src_port:      int32
    dst_ip:        string
    dst_port:      int32
    protocol:      string
    dns_query:     string
    ja3:           string
    ja3s:          string
    bytes_sent:    int64
    bytes_recv:    int64
  }
  registry?: { ... }
  auth?: { ... }
  script?: { content: string, interpreter: string }

  // 富化字段（Ingestion Gateway 填充）
  enrichment: {
    geo:           GeoInfo
    threat_intel:  ThreatIntelMatch[]
    mitre_ttps:    string[]      // ["T1059.001", "T1055.012"]
    risk_score:    float         // 0.0 - 100.0
  }
}
```

### 7.2 响应能力矩阵

| 响应动作 | 延迟要求 | 审批要求 | 可回滚 | 实现方式 |
| --- | --- | --- | --- | --- |
| 进程终止 | ≤ 3s | 自动/手动 | 否 | Agent kill -9 / TerminateProcess |
| 文件隔离 | ≤ 5s | 自动/手动 | 是 | 移至加密隔离区 |
| 网络隔离 | ≤ 3s | 需审批 | 是 | Agent 注入防火墙规则，仅保留 C2 通道 |
| 注册表回滚 | ≤ 5s | 手动 | 是 | 基于变更日志还原 |
| 用户会话锁定 | ≤ 10s | 需审批 | 是 | 强制注销 + 禁用账户 |
| 远程取证 | 按需 | 需审批 | N/A | 内存 Dump + 关键文件收集 |
| 远程 Shell | 按需 | 双人审批 | N/A | 加密终端，全程录屏+审计 |
| 自动 Playbook | ≤ 10s | 预审批 | 部分 | 编排引擎自动执行 |

### 7.3 自动化 Playbook 示例（Ransomware Response）

```yaml
name: ransomware-auto-response
trigger:
  detection_rule: "ransomware-*"
  min_severity: HIGH
  min_confidence: 0.85

stages:
  - name: immediate-containment
    parallel: true
    actions:
      - type: kill_process
        target: "{{ detection.process.pid }}"
      - type: network_isolate
        target: "{{ detection.agent_id }}"
        allow: ["edr-c2-channel"]
      - type: file_quarantine
        target: "{{ detection.file.path }}"

  - name: scope-assessment
    actions:
      - type: ioc_sweep
        iocs:
          - "{{ detection.file.hash }}"
          - "{{ detection.network.dst_ip }}"
        scope: "all-endpoints"
        timeframe: "72h"

  - name: notification
    actions:
      - type: create_incident
        severity: CRITICAL
        assign_to: "soc-tier2"
      - type: notify
        channels: ["slack:#soc-critical", "pagerduty:oncall"]
        message: |
          🚨 Ransomware detected on {{ detection.host.hostname }}
          Process: {{ detection.process.cmdline }}
          File: {{ detection.file.path }} ({{ detection.file.hash }})
          Action taken: Process killed + Host isolated

  - name: forensic-collection
    requires_approval: true
    approvers: ["soc-lead", "ir-manager"]
    actions:
      - type: memory_dump
        target: "{{ detection.agent_id }}"
      - type: collect_artifacts
        artifacts:
          - "$MFT"
          - "prefetch"
          - "event_logs"
          - "browser_history"
          - "registry_hives"
```

## 九、安全性设计

### 9.1 数据安全

| 环节 | 措施 |
| --- | --- |
| 传输加密 | TLS 1.3 + mTLS（Agent ↔ Gateway），服务间 mTLS（Istio） |
| 存储加密 | AES-256-GCM（静态数据），LUKS（磁盘级） |
| 密钥管理 | Vault 自动轮换，Agent 证书有效期 90 天 |
| 数据脱敏 | PII 字段（用户名/IP）支持伪匿名化存储 |
| 数据隔离 | 多租户逻辑隔离（Tenant ID 贯穿全链路），关键场景物理隔离 |

### 9.2 平台自身安全

- **Supply Chain**：Agent 二进制签名验证，更新包完整性校验 (Sigstore/TUF)
- **Zero Trust**：所有内部服务间 mTLS，无隐式信任
- **Audit Trail**：管理面所有操作全量审计，不可篡改（append-only log）
- **Penetration Testing**：季度外部渗透测试 + 持续红队演练
- **Agent 安全**：Agent 自身作为高价值目标，需抗 rootkit 级对抗
